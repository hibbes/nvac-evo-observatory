# Findings from the first wedge (multi-agent analysis, 2026-06-24)

Source: `events/wedge-20260624T071803.txt` + the baseline transition rows.
Full machine output: `wedge-analysis-2026-06-24.json`. This is n=1, so everything
below is a hypothesis with a confidence level, not a settled conclusion.

## Register-confirmed (high confidence)

- It is a HARDWARE FETCH stall, not a software submission stall. GET freezes
  behind PUT while PUT keeps advancing: core_get pinned 0x2c8 / core_put 0x308;
  base1_get pinned 0x8ac while base1_put creeps 0x91c -> 0x9ac (the compositor
  keeps queuing flips into a dead channel). A software stall would freeze PUT.
- bit31 of the channel control reg = "channel busy/processing", decoded from the
  driver's own init/fini wait loops (nv50_disp_dmac_init writes 0x13 then waits
  for bit31 to clear). In the wedge bit31 is latched set forever on core
  (0x610200) and base1 (0x610220): exactly what nv50_dmac_wait / the notifier
  time out on.
- Only the active channels wedge. base0 (idle head) stays 0x4003001b, bit31 never
  sets. Not a blanket all-channels power event.
- PMC stays powered (0x000200 bit30 set) the whole time: a latched internal fetch
  state, which is why only a PDISPLAY power-cycle clears it and a per-channel
  re-arm cannot.

## Trigger correlate (medium confidence)

- The wedge fires on the FIRST display commit after ~30 s of drained idle
  (core_put steps 0x200 -> 0x2e8 at the onset sample).
- At that exact sample, 0x61002c bit3 goes 0 -> 1 = head1 vblank IRQ re-enabled
  (drm_crtc_vblank_get). (This register is the DISP interrupt-enable; the v1
  sampler mislabeled it "super1".) So a vblank-synced head1 flip was committed
  right when it wedged.
- Leading hypothesis: the wedge fires on the first vblank-synced flip after an
  idle/wake period, with a core EVO UPDATE that never settles dragging the
  interlocked head1 base flip into the fetch-park.

## Honest caveats (what n=1 cannot show)

- "Core parks first" (dmesg core-notifier before base-1) may be a DRIVER
  FLUSH-ORDER ARTIFACT: commit_tail always waits the core notifier before the
  base waits, so the ordering is structurally forced regardless of which actually
  stalled first. The register margin is one 5 s sample. Do not over-claim core
  causation.
- bit3 toggles on every commit-after-idle, including the thousands that do NOT
  wedge. Without fine-cadence captures of healthy wakes, the idle->commit trigger
  is unfalsified.
- A shared fetch arbiter is inferred, not observed.
- An EVO method/TRAP error may be the real first-mover and was not sampled in v1.

## Acted on (sampler v2)

- Now samples the per-channel error latches (0x610080/84 core, 0x610090/94
  base1), the real supervisor reg (0x610030), and the correctly labelled DISP

## Still needed

  always latch with/before base1; can base1 park alone; is the vblank re-arm
  always present).
- 100 ms symmetric burst around the onset (pre + post) to settle core-vs-base.
- drm.debug=0x1e (DRM_UT_KMS) so the commit log survives the wedge.
- Continuous compositor / wlopm / swayidle context aligned to the register
  timeline, to confirm or refute "first commit after the monitor self-slept".
